# When a render-verify harness drives a React CONTROLLED input (`<input type="color">`, text, etc.), setting `el.value` directly does NOT fire React's onChange (React tracks the value via its own setter, so state never updates and the test wrongly FAILS) — instead call the prototype setter `Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype,"value").set.call(el, v)` then `dispatchEvent(new Event("input",{bubbles:true}))` (Playwright `fill()`/`type()` already do this for text but NOT for color inputs), so the simulated change is indistinguishable from a real user edit and the assertion reflects the real product behavior.
