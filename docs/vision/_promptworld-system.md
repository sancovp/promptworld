# vision: PromptWorld — the system (aggregated from the journal, 2026-05-31)

Cross-cutting vision for PromptWorld. Everything here is IDEA/DECISION, not yet built (impl goes to
doc(m) when it exists). Source: the 2026-05-31 journal design arc (conceived inside the doc-mirror
system; this is the projection that should have happened live).

## What PromptWorld IS
A **\*World** in the JobWorld / HealthWorld family — same implementation pattern, new domain. Its domain
is **compiling agents**. (JobWorld runs a business; HealthWorld runs a diagnostic sim over health data;
PromptWorld compiles agents.) It is built **on cave-teams** (the existing \*World substrate), as a fork
of the JobWorld/HealthWorld pattern — NOT a from-scratch server (an early reinvention as a bare FastAPI
app was the wrong path; the real pattern is cave-teams).

  (file-based routing). PromptWorld = a fork of this.
- [v4]  **PromptWorld** — the \*World itself: the team of compiler-agents + the interface.

## The interface (corrected understanding)
The interface is **NOT an in-browser shell**. It is **a dashboard (FastAPI + WebSocket — cave-teams
"Phase 3: wrap the team in HTTP routes") that IS the live interface to the running team in the
container.** The user does **everything** on the interface; main Claude can also drive it via the API.

## Runtime + auth (the hard-won correction)
- [v5]  PromptWorld runs **in a container**, talked to **remotely**.
- [v6]  **Auth is a user action, not a build action.** A fresh container is NOT authenticated by copying
  `credentials.json` (that never works — it was the cause of every SDK `initialize` timeout). The user
  authenticates by running **`claude setup-token`** (or `claude`) themselves, through the interface, in
  the live container shell the interface exposes. The build does nothing for auth.
- [v7]  A **skill starts the system** (the skill "has the server inside it").
- [v8]  Do NOT: spawn fresh containers + cp creds per agent run; run it on the host as the deliverable.

## SDK facts learned (apply when building the real runtime)
- [v9]  `ClaudeAgentOptions` must pin a current `model` (default `claude-3-5-sonnet-latest` 404s on this
  subscription → use `claude-sonnet-4-6`).
- [v10]  Lean options (no heavy `setting_sources` loading the host's 105 skills + MCPs) or `initialize` stalls.
- [v11]  `ClaudeSDKClient` gives real multi-turn chat (`query` + `receive_response`); persistent client = the session.

## Example vs final form
PromptWorld (a legible, hand-assembled team of explicit compiler-agents you wire together) is the
**EXAMPLE / dumb-version**. The **final form is SANCREV OPERA**: one agent that *deduces* what can and
should be done over the option-tree about agents. "What CAN be done" = SOMA admissibility (the mereo
validator — the terminus of the guarantee stack); "what SHOULD" = the OPERA deduction. The option-tree
itself = the space paia-builder's specs enumerate.

## Why many persona-scoped agents (not one)
A guy is a closed chain that knows its identity. "Be everything" breaks closure (the one-agent can't
hold its identity; progressive disclosure over all of it is too expansive). So PromptWorld is many
persona-scoped agents (each its own doc-mirror identity), coordinated — talk-to-one standalone, or a CEO
runs the team. The single deducing agent is the final form, for later.
